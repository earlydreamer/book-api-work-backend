## 1. 이 문서의 목적

이 문서는 `today-us-front`를 본 구현 전에 다시 정렬하기 위한 기준선이다.

현재 프론트는 `화면 흐름을 빠르게 보는 단일 목업`으로는 충분히 역할을 했다.
하지만 지금 상태로 API 계약과 실제 연동으로 들어가면, 화면은 예쁜데 제품 상태가 서로 다른 규칙으로 움직이는 문제가 생긴다.

이번 정렬의 목적은 세 가지다.

1. 제품 상태를 한 문장으로 설명 가능한 수준으로 단순화한다.
2. 각 화면이 어떤 상태에서 무엇을 보여줘야 하는지 고정한다.
3. mock 데이터가 화면 데모를 위한 임시 장치가 아니라, 본 구현 전 계약 검증 장치가 되도록 바꾼다.

## 2. 지금 상태에서 실제로 어긋난 것

### 2.1 상태 규칙이 페이지마다 다르다

- 피드는 신규 유저여도 이미 기록이 쌓인 커플처럼 보인다.
- 책 만들기 진입 조건이 화면마다 다르다.
- 로그인 상태가 새로고침 이후 유지되지 않는다.
- 연결 상태와 기록 상태와 주문 상태가 서로 독립적으로 정의되지 않았다.

### 2.2 mock 데이터가 제품 계약을 오염시킨다

- `globalMockMoments` 하나가 모든 화면의 truth처럼 쓰인다.
- 신규 유저, 연결 대기 유저, 활성 커플, 주문 가능 커플을 같은 데이터로 흉내낸다.
- 그 결과, 화면이 보여주는 감정선과 제품 단계가 꼬인다.

### 2.3 디자인 시스템 문서와 실제 화면의 우선순위가 다르다

- 랜딩은 아직 템플릿형 teaser 구성이 남아 있다.
- 피드는 일일 기록보다 책 만들기 신호가 더 크게 들린다.
- 한국어 감성 제품인데 영어 라벨이 섞여 있다.
- alert, confirm 같은 브라우저 기본 UI가 제품 톤을 끊는다.

## 3. 이번 정렬에서 잠글 결정

### 3.1 가장 중요한 제품 원칙

- 이 앱의 1차 목적은 `매일 기록하게 만드는 것`이다.
- 책 만들기는 차별점이지만, 일일 기록 루프를 이기면 안 된다.
- 부분 카드는 실패가 아니라 유효한 기록이다.
- 온보딩 종료 기준은 `회원가입 완료`가 아니라 `첫 카드 생성`이다.
- 연결 전 유저에게는 아카이브 성취감이 아니라 `파트너 연결`과 `첫 기록 시작`이 핵심 경험이다.

### 3.2 책 주문 가능 기준

책 주문 가능 기준은 기존 제품 명세를 따른다.

- 기준: `최근 30일 중 기록 20일 이상`
- 기준 출처: `docs/specs/2026-04-05-today-our-mvp-v2.md`
- 따라서 프론트의 `15개`, `30개`, `60개` 같은 임시 숫자는 모두 제거 대상이다.

### 3.3 책 화면 정책

`/make-book`은 더 이상 `주문 가능 상태에서만 들어가는 화면`이 아니다.

이 화면은 두 구간으로 나뉜다.

1. `주문 전 성장 구간`
- 지금 얼마나 쌓였는지
- 어떤 책이 될지
- 무엇이 더 필요하고 왜 아직 주문할 수 없는지

2. `주문 실행 구간`
- 스냅샷 생성
- 조판 진행
- 배송지 입력
- 결제

즉, 주문 가능 이전에도 `책`은 보여주되, `주문 플로우`는 잠겨 있어야 한다.

### 3.4 피드 우선순위

피드의 1번 목적은 항상 `오늘 카드 상태`를 보여주는 것이다.

우선순위는 아래 순서를 따른다.

1. 오늘 기록 액션
2. 파트너 연결 상태
3. 오늘 카드 완성 여부
4. 최근 기록
5. 책으로 성장 중인 진행 상태

책 관련 CTA는 이 순서를 절대 넘지 않는다.

## 4. 표준 상태 모델

프론트는 앞으로 페이지별 임시 분기 대신, 아래 4축 상태를 조합해서 화면을 만든다.

### 4.1 상태 축

| 축 | 값 |
|---|---|
| 인증 상태 | `visitor`, `authenticated` |
| 연결 상태 | `unconnected`, `invite-pending`, `connected` |
| 오늘 카드 상태 | `empty`, `mine-only`, `partner-only`, `complete`, `closed` |
| 책 상태 | `growing`, `eligible`, `snapshot-building`, `ready-to-order`, `ordered` |

### 4.2 파생 UI 상태

```text
visitor
  -> 랜딩

authenticated + unconnected
  -> 연결 온보딩

authenticated + invite-pending + today empty
  -> 피드 첫 기록 유도 + 초대 코드 강조

authenticated + connected + today empty
  -> 오늘 기록 CTA 강조

authenticated + connected + today mine-only
  -> 파트너 대기 상태 + nudging

authenticated + connected + today complete
  -> 오늘 카드 완성 상태 + 최근 기록 요약

book growing
  -> 책 성장 화면

book eligible
  -> 스냅샷 생성 CTA 노출

snapshot-building
  -> 진행 상태

ready-to-order
  -> 배송/결제

ordered
  -> 주문 완료 + 내 공간 주문 추적
```

### 4.3 파생 상태 계산 원칙

- 페이지 내부에서 매직 넘버로 계산하지 않는다.
- `isLoggedIn`, `isConnected`, `partnerName`, `globalMockMoments.length`를 섞어서 직접 UI를 만들지 않는다.
- 공통 selector 또는 adapter가 `homeState`, `bookState`, `todayCardState`를 계산한다.

## 5. 화면별 계약

## 5.1 랜딩

목적:
- `Today Us가 무엇인지`
- `왜 둘만의 기록이 의미 있는지`
- `첫 액션이 무엇인지`

보여줘야 할 것:
- 제품 한 줄 가치
- 프라이빗함
- 기록이 책으로 이어지는 미래
- primary CTA 1개

보여주지 말 것:
- 앱 내부 정보 구조 설명
- 기능 나열형 teaser 3개
- 가짜 운영 링크

결정:
- 현재 랜딩의 3열 teaser는 제거 또는 축소한다.
- 랜딩은 `editorial hero + 한 줄 증거 + 첫 행동` 구조로 단순화한다.

## 5.2 로그인 / 회원가입

목적:
- 빠르게 계정을 만든다.
- 브라우저 기본 alert 없이 현재 상태를 이해시킨다.

보여줘야 할 것:
- 진행 단계
- 이메일 인증 상태
- 비밀번호 불일치 상태
- 완료 가능 여부

결정:
- `alert()` 대신 인라인 상태 메시지 사용
- 성공/실패/인증 완료 모두 폼 안에서 설명
- CTA는 한 화면에 1개 강하게 유지

## 5.3 연결하기

목적:
- 파트너 연결을 자연스럽게 끝내게 한다.

상태:
- 초대 코드 있음
- 내 공간 먼저 만들기
- 초대 대기 중
- 연결 완료

결정:
- `join`, `create`, `invite-pending`을 별도 시나리오로 분리한다.
- 연결 완료 직후 바로 `아카이브 성취`를 보여주지 않는다.
- 연결 이후 첫 피드는 `첫 기록 남기기`가 중심이다.

## 5.4 피드

목적:
- 오늘 카드 상태를 설명하고, 오늘 액션을 일으킨다.

상태별 1순위:

| 상태 | 피드 상단 1순위 |
|---|---|
| 연결 전 | 초대 코드 공유 + 첫 기록 시작 |
| 연결 후, 오늘 비어 있음 | 오늘 기록 남기기 |
| 나만 기록함 | 파트너 대기 + 필요 시 부드러운 nudging |
| 둘 다 기록함 | 오늘 카드 완성 + 최근 기록 |

결정:
- `이번 달 우리의 기록`은 보조 정보다.
- `책으로 엮을 수 있어요`는 `bookState === eligible`일 때만 강하게 노출한다.
- 연결 전에는 책 관련 강한 CTA를 숨기고, 진행 정보만 최소 노출한다.
- 새 관계가 시작된 뒤에는 이전 관계 기록을 피드에 노출하지 않는다.
- 피드는 항상 현재 연결 cycle의 기록만 읽는다.

## 5.5 아카이브

목적:
- 기록이 쌓이고 있다는 감각을 준다.

결정:
- 월별 구분은 유지한다.
- 부분 카드와 완성 카드를 같은 가치로 다룬다.
- 연결 전 상태에서는 아카이브 전체보다 샘플 또는 빈 상태가 맞다.
- 보관함은 `현재 연결 기록`과 `이전 연결 기록`을 별도 섹션으로 나눈다.
- 이전 연결 기록 카드에는 `이전 파트너와의 기록` 라벨을 붙인다.
- 이전 연결 기록은 현재 관계 진행도와 섞어 계산하지 않는다.
- 연결을 끊어도 기록은 삭제하지 않고 보관함으로 이동한다.
- 다시 연결하면 현재 기록 수와 책 진행도는 0일부터 다시 시작한다.

## 5.6 기록

목적:
- 20~30초 안에 하루 조각을 남긴다.

결정:
- 사진 또는 무드 카드 중 하나는 남길 수 있어야 한다.
- 감정은 필수다.
- 저장 완료 뒤에는 피드 이동만이 아니라 `오늘 카드에 어떻게 반영되는지`가 보여야 한다.
- 위치, 태그는 MVP 본 구현에서 비활성 또는 숨김 처리한다. 장식 버튼으로 두지 않는다.

## 5.7 책 만들기

목적:
- 기록이 책으로 자라는 경험을 설명한다.
- 주문 가능 상태에서는 실제 주문 흐름으로 자연스럽게 이어진다.

상태:

| 상태 | 화면 목적 |
|---|---|
| growing | 진행률, 미리보기, 주문 조건 설명 |
| eligible | 스냅샷 생성 CTA |
| snapshot-building | 진행 상태 |
| ready-to-order | 배송지/결제 |
| ordered | 주문 완료 |

결정:
- `기록 부족 시 잠금 화면` 하나로 끝내지 않는다.
- 주문 가능 전에도 디지털 미리보기와 성장 설명이 있어야 한다.
- 선택 리스트, 조판 진행, 결제는 `eligible` 이후에만 열린다.

## 5.8 내 공간

목적:
- 관계 상태와 개인 설정을 관리한다.

결정:
- 연결 상태, 알림, 프라이버시, 주문 내역을 분리한다.
- mock 단계에서도 컨텍스트 외부의 `localStorage` 직접 조작은 금지한다.
- `파트너 끊기`는 기록 삭제가 아니라 `현재 연결 종료 + 기록 보관`으로 정의한다.
- 재연결 시 현재 연결 cycle은 새로 시작한다.

## 7. mock 데이터 전략 교체

### 7.1 금지

- 모든 화면이 `globalMockMoments` 하나를 공유하는 구조
- 신규 유저와 헤비 유저를 같은 fixture로 표현하는 구조
- 페이지가 직접 날짜/개수/자격 조건을 계산하는 구조

### 7.2 새 전략

fixture는 시나리오 단위로 분리한다.

```text
scenario/visitor
scenario/new-user
scenario/invite-pending
scenario/connected-empty
scenario/today-partial
scenario/today-complete
scenario/book-growing
scenario/book-eligible
scenario/order-ready
```

각 fixture는 아래 shape를 가진다.

```text
session
relationship
homeSummary
todayCard
archiveSummary
bookSummary
settingsSummary
```

즉, 프론트는 더 이상 raw card array만 받지 않는다.
화면이 바로 필요한 view model을 adapter가 내려주는 구조로 바꾼다.

## 8. 프론트 mock adapter 계약

본 구현 전 프론트는 아래 adapter 계층을 반드시 거친다.

### 8.1 필요한 adapter

- `authAdapter`
- `relationshipAdapter`
- `homeAdapter`
- `recordAdapter`
- `archiveAdapter`
- `bookAdapter`
- `settingsAdapter`

### 8.2 규칙

- 페이지는 adapter만 호출한다.
- 페이지는 `globalMockMoments`를 직접 import하지 않는다.
- 숫자 기준과 상태 문자열은 selector/constants 한 곳에서만 관리한다.
- 새로고침 후에도 현재 시나리오가 복원되도록 mock session persistence를 둔다.

---

## 9. 본 구현 전 Definition of Ready

아래가 만족되기 전에는 실제 API 연동 단계로 넘어가지 않는다.

1. 표준 상태 모델이 문서와 코드에서 동일하다.
2. 피드, 연결, 기록, 책, 내 공간이 시나리오 fixture 기반으로 렌더링된다.
3. 책 자격 기준이 한 곳에서만 계산된다.
4. 새로고침 이후 로그인/연결/mock 시나리오가 유지된다.
5. 모든 주요 CTA가 한국어로 통일된다.
6. `alert`, `confirm`, 가짜 placeholder 링크가 핵심 플로우에 남아 있지 않다.
7. 최소 테스트가 있다.
   - 인증 복원
   - 연결 상태 분기
   - 오늘 카드 상태 분기
   - 책 자격 계산
   - 연결 해제/재연결 후 기록 보존 정책
   - 피드의 현재 관계 전용 노출 정책
   - 보관함의 현재/이전 연결 구분 정책

## 10. 이번 정렬에서 의도적으로 안 하는 것

- 실제 백엔드 API 연동
- 실제 결제 위젯 연동
